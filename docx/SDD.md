## 1. 设计目标

本系统的设计目标不是实现某一个固定模型，而是构建一套完整的轴承寿命预测实验框架。详细设计的重点包括：

1. 统一真实数据集的表示方法。
2. 将预处理、特征提取、标注、训练、预测、评估、可视化和实验记录解耦。
3. 允许用户在不破坏主流程的前提下接入新算法模块。
4. 支持课程项目的代码演示、测试验证和文档交付。

系统的设计原则如下：

1. 单一职责：每个模块只解决一类问题。
2. 低耦合：数据表示与模型训练分离，训练流程与回调分离。
3. 高内聚：同类能力集中到一个包中管理。
4. 可复现：实验配置、日志、结果和模型都可导出。
5. 易扩展：通过统一接口和注册机制降低新增组件成本。

## 2. 系统架构设计

### 2.1 总体架构

系统总体采用分层式与组件化结合的架构：

```text
数据接入层
  ↓
数据抽象层
  ↓
预处理与特征工程层
  ↓
标注与任务构造层
  ↓
模型训练与测试层
  ↓
预测与不确定性分析层
  ↓
评估与可视化层
  ↓
实验记录与产物导出层
```

这种架构的核心好处在于：

1. 不同数据集先统一为 `BearingEntity`，上层不必关心目录结构差异。
2. 不同任务通过不同 `Labeler` 构造数据集，训练器本身不和任务逻辑耦合。
3. 不同模型通过统一模型接口接入。
4. 训练附加能力全部通过回调实现，避免训练器过于臃肿。

### 2.2 目录结构与职责映射

| 目录 | 主要职责 |
| --- | --- |
| `common` | 注册器、序列化、模型文件保存等公共能力 |
| `data` | 实体对象、窗口数据集对象、合成数据生成 |
| `dataset` | 真实数据集加载器与数据资源描述 |
| `preprocess` | 归一化、裁剪、切窗、阶段策略等 |
| `feature` | 时域、频域特征提取与健康指标构造 |
| `labeling` | 面向任务的数据集构造逻辑 |
| `models` | CNN / RNN / Transformer / MLP 等模型 |
| `training` | 训练器、测试器、回调、实验记录 |
| `prediction` | 直接预测、滚动预测、不确定性预测 |
| `evaluation` | 指标实现与评估器 |
| `visualization` | 图像生成 |
| `tests` | 自动化测试 |
| `docs` | 文档交付物 |

## 3. 核心数据结构设计

### 3.1 BearingEntity

`BearingEntity` 是系统最重要的数据抽象，用于统一表示某个轴承在多个时间快照上的原始观测。

#### 设计原因

真实数据集在目录层次、文件命名、字段数量和附加信息上存在差异。如果直接在训练流程里处理原始目录和文件，会导致上层逻辑复杂、耦合严重。因此系统在数据接入后第一时间转换为统一实体对象。

#### 主要字段

| 字段 | 说明 |
| --- | --- |
| `entity_id` | 轴承编号，如 `Bearing1_1` |
| `dataset_name` | 数据来源名称 |
| `samples` | 快照级数据表 |
| `sample_rate` | 采样率 |
| `metadata` | 附加信息，如工况、路径、分组等 |

#### 设计要点

1. `samples` 中每一行表示一个时间快照。
2. 通道列以数组形式保存，如水平振动、垂直振动、温度序列。
3. `channel_names()` 只返回真正的数组型通道，不暴露元数据列。

### 3.2 BearingWindowDataset

`BearingWindowDataset` 是面向模型输入的统一数据结构，兼容 PyTorch `Dataset`。

#### 主要字段

| 字段 | 含义 |
| --- | --- |
| `inputs` | 模型输入矩阵或序列 |
| `targets` | 监督目标 |
| `metadata_frame` | 样本级元数据 |
| `task_type` | 回归 / 分类 |
| `target_name` | 目标字段名 |
| `input_name` | 输入表示名 |
| `feature_frame` | 可选特征表 |
| `extra_targets` | 附加监督信号 |

#### 设计要点

1. 既能用于原始信号模型，也能用于特征向量模型。
2. 支持时间顺序切分。
3. 支持导出为 CSV / PKL 方便调试和报告分析。

### 3.3 ExperimentConfig 与实验产物

训练过程会将核心配置写入实验目录。设计上将“实验是一个一等对象”，而不是“顺手打印一下参数”。这样做的目的有两点：

1. 支持复现实验。
2. 支持课程答辩中解释模型配置来源。

配置内容包括：

1. 数据集名称。
2. 模型名称和超参数。
3. 学习率、权重衰减、批大小、最大轮次。
4. 采样策略、预测模式。
5. 回调配置、预处理配置。

## 4. 数据接入层设计

### 4.1 BaseBearingLoader

`BaseBearingLoader` 负责封装数据集遍历、文件排序、文件解析和实体装配的共性逻辑。

#### 设计思想

1. 使用模板方法模式。
2. 公共逻辑放在基类中。
3. 数据集差异通过子类重写局部方法处理。

#### 核心行为

1. `list_entities()`：枚举全部实体。
2. `load_entity(entity_id)`：按实体编号加载。
3. `_iter_signal_files()`：决定读取哪些文件。
4. `_read_signal_file()`：将不同 CSV/TXT 文件统一转为数值表。
5. `_build_entity_metadata()`：补充数据集特有元数据。

#### 当前真实数据兼容设计

1. 对 XJTU-SY，CSV 允许带表头，读取后会强制数值化并丢弃非数值行。
2. 对 PHM2012，主时间轴由 `acc_*.csv` 决定，`temp_*.csv` 按快照编号对齐为 `Temperature` 通道。

### 4.2 XJTULoader

`XJTULoader` 用于读取 XJTU-SY 轴承退化数据。

#### 特殊处理

1. 根据工况目录推断运行条件。
2. 支持目录递归扫描。
3. 兼容带表头的两列振动数据文件。

### 4.3 PHM2012Loader

`PHM2012Loader` 用于读取 IEEE PHM 2012 / FEMTO 数据。

#### 特殊处理

1. 保留 `acc_*.csv` 对应的水平/垂直振动通道。
2. 对齐 `temp_*.csv` 作为温度通道，而不是简单忽略。
3. 在元数据中记录所属数据分组，如 Training / Validation / Test。

#### 设计理由

温度虽然不是主训练通道，但是真实工业场景中的重要观测量。保留温度可以支持：

1. 多模态建模。
2. 可视化与解释性分析。
3. 后续将温度作为附加特征使用。

## 5. 预处理与特征工程设计

### 5.1 预处理流水线

预处理层采用“变换器顺序组合”的设计。当前典型流程为：

```text
原始信号
→ RobustClip
→ Normalize
→ SlidingWindowSegmenter
```

#### 设计理由

1. 保持每一步职责清晰。
2. 便于替换某个变换器。
3. 便于实验日志中记录完整预处理路径。

### 5.2 滑动窗口设计

滑动窗口是连接原始长序列与模型输入的关键步骤。

主要参数：

1. `window_size`
2. `stride`

设计要求：

1. 支持重叠窗口。
2. 保持时间顺序。
3. 能与回归、分类和滚动预测三类任务兼容。

### 5.3 特征提取设计

`SignalFeatureExtractor` 同时提取时域和频域特征。

#### 时域特征

1. 均值
2. 方差
3. 均方根
4. 峰值
5. 峭度
6. 偏度

#### 频域特征

1. 主频
2. 频域能量
3. 谱熵

#### 健康指标构造

健康指标并不是固定物理量，而是从多特征中构造出的退化强度表示。设计上将其作为独立结果输出，便于：

1. 阶段划分。
2. 滚动预测。
3. 退化趋势可视化。

## 6. 标注层设计

标注层是系统中最重要的“任务适配层”，负责把 `BearingEntity` 转换为可训练数据集。

### 6.1 BearingRulLabeler

用于构造 RUL 回归数据集。

#### 输入

1. `BearingEntity`
2. 通道名
3. 窗口参数
4. 输入表示类型

#### 输出

`BearingWindowDataset`

#### 设计特点

1. 可输出原始信号窗口。
2. 可输出特征向量。
3. 目标值为快照级 RUL。

### 6.2 BearingStageLabeler

用于构造退化阶段分类数据集。

#### 处理过程

1. 对每个快照切窗。
2. 对每个快照构造健康指标。
3. 调用阶段策略生成阶段标签。
4. 形成分类样本。

### 6.3 HealthIndicatorLabeler

用于构造健康指标序列预测数据集。

#### 设计用途

当研究目标从“直接预测 RUL”转向“预测退化曲线未来形态”时，需要针对健康指标序列构建专用样本。本模块就是面向该需求设计。

## 7. 模型层设计

### 7.1 统一模型设计思想

系统支持多种模型，但接口设计保持统一。统一的好处在于：

1. 训练器不需要区分具体模型类型。
2. 测试器和预测器可以复用。
3. 用户可以快速横向对比不同模型。

### 7.2 CNN

适合直接处理振动信号窗口，能够自动学习局部模式和频率相关结构。

适用场景：

1. 端到端 RUL 回归。
2. 原始信号阶段分类。

### 7.3 RNN

适合建模时间顺序依赖，尤其适用于序列特征或健康指标序列。

### 7.4 Transformer

适合处理较长序列和注意力分布分析，支持后续生成注意力热图。

### 7.5 MLP

适合处理已经提取好的固定长度特征向量，是轻量基线模型的重要组成部分。

## 8. 训练与测试层设计

### 8.1 BaseTrainer

`BaseTrainer` 是统一训练入口。

#### 主要职责

1. 构建 DataLoader。
2. 组织训练循环。
3. 执行前向、反向与优化更新。
4. 在适当阶段触发回调。
5. 记录训练结果与最佳轮次。

#### 设计重点

训练器本身不直接硬编码 EarlyStopping、TensorBoard 或报警逻辑，而是只提供钩子时机：

1. `on_train_start`
2. `on_epoch_start`
3. `on_batch_end`
4. `on_validation_end`
5. `on_epoch_end`
6. `on_train_end`

### 8.2 Callback 体系

Callback 设计的核心目的是让训练附加能力插件化。

#### 内置回调

1. `EarlyStopping`
2. `TensorBoardCallback`
3. `GradientAlertCallback`
4. `ExperimentLoggerCallback`

#### 优势

1. 训练器职责更纯粹。
2. 新的监控或日志逻辑可独立新增。
3. 课程演示中可以清楚解释“训练主流程”和“附加监控能力”的边界。

### 8.3 BaseTester

测试器统一负责：

1. 调用模型推理。
2. 支持可插拔预测器。
3. 输出 `TestResult`。
4. 兼容回归、分类和不确定性结果。

## 9. 预测层设计

### 9.1 DirectPredictor

直接基于输入样本给出预测值，是默认推理方式。

### 9.2 RollingPredictor

用于根据历史序列递推多个未来时间点的预测值。

#### 应用场景

1. 健康指标未来趋势分析。
2. 单步模型扩展为多步预测。

### 9.3 MonteCarloDropoutPredictor

通过多次随机 Dropout 前向传播估计预测均值和不确定性。

#### 设计价值

1. 让系统不只输出单点预测。
2. 可以回答“模型对当前预测有多确定”。
3. 更接近工程预警系统的需求。

---

## 10. 评估与可视化设计

### 10.1 Evaluator

`Evaluator` 用于聚合多个指标实例，统一评估接口。

#### 设计动机

不同实验往往关注不同指标。如果每次手动编写指标计算逻辑，会导致重复代码和不一致输出。`Evaluator` 通过组合模式解决这个问题。

### 10.2 指标体系

当前指标可分为三类：

1. 基础回归指标：MAE、MSE、RMSE、MAPE、PercentError。
2. 挑战赛风格指标：PHM2012Score、PHM2008Score、NASAScore。
3. 分类指标：Accuracy。

### 10.3 可视化设计

`ResultVisualizer` 当前支持：

1. 预测结果曲线。
2. 退化阶段图。
3. 混淆矩阵。
4. 注意力热图。

#### 设计说明

这些图并非仅用于“好看”，而是直接服务于课程答辩：

1. 预测曲线说明模型拟合趋势。
2. 退化阶段图说明阶段划分策略是否合理。
3. 混淆矩阵说明分类边界的误差分布。
4. 注意力热图说明模型关注区域。

## 11. 实验记录与导出设计

### 11.1 ExperimentTracker

`ExperimentTracker` 负责创建实验目录并导出产物。

#### 产物类型

1. 配置文件。
2. 历史记录文件。
3. 指标文件。
4. 预测结果文件。
5. 报警文件。
6. 模型文件。
7. TensorBoard 日志目录。
8. 可视化图片。

### 11.2 文件格式设计

| 类型 | 格式 |
| --- | --- |
| 配置 | JSON / YAML |
| 历史 | CSV / PKL |
| 预测结果 | CSV / PKL |
| 指标 | JSON / YAML |
| 模型 | PT |
| 图片 | PNG |

#### 设计理由

1. `CSV` 便于直接查看和报告处理。
2. `PKL` 保留 Python 对象结构。
3. `JSON/YAML` 便于配置记录和跨工具查看。
4. `PT` 适合保存 PyTorch 模型权重。

## 12. 错误处理与鲁棒性设计

### 12.1 数据读取错误

系统对以下问题给出明确异常：

1. 实体目录不存在。
2. 快照文件为空。
3. CSV 解析失败。
4. 通道缺失。

### 12.2 训练过程异常

训练过程可能出现：

1. 梯度爆炸。
2. 梯度消失。
3. 输入维度不匹配。
4. 验证集为空。

设计上通过回调记录报警，并尽量将问题暴露为可定位的异常。

### 12.3 真实数据兼容问题

从真实数据测试中，系统已经明确处理以下兼容问题：

1. XJTU-SY 文件带表头。
2. PHM2012 同时包含振动和温度文件。
3. 真实数据目录层次与示例目录不完全一致。

## 13. 性能与扩展性设计

### 13.1 性能考虑

1. 默认保持 CPU 可运行。
2. 真实数据测试时允许只抽取子集做 smoke test。
3. 日志和图像导出安排在训练后或回调中进行，避免主训练循环过重。

### 13.2 扩展点设计

系统预留的主要扩展点包括：

1. 新数据集加载器。
2. 新预处理器。
3. 新阶段策略。
4. 新模型。
5. 新预测器。
6. 新指标。
7. 新回调。

新增模块时应遵循的原则：

1. 优先复用现有实体和数据集结构。
2. 不破坏统一 API。
3. 新能力优先通过扩展实现，而不是修改主流程。

---

## 14. 设计总结

本系统详细设计的核心不是单点算法复杂度，而是围绕“真实数据 + 多任务 + 可复现 + 可答辩”构建完整工程链路。系统通过统一实体、解耦标注层、统一训练器和可插拔回调，将课程项目常见的“脚本堆叠式实现”提升为真正可维护的实验框架。

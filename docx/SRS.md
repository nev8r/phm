## 1. 引言

### 1.1 编写目的

本文档用于系统化描述“工业轴承故障预测系统”的软件需求，为项目开发、联调测试、课程验收、实验复现和后续扩展提供统一依据。本文档关注的问题不是单一算法实现，而是围绕真实工业轴承数据构建一套可运行、可扩展、可记录、可评估、可展示的预测软件系统。

本文档的主要目标如下：

1. 明确系统的业务目标、边界和使用场景。
2. 给出功能需求与非功能需求，避免开发范围模糊。
3. 为详细设计文档、测试文档和答辩材料提供需求基线。
4. 为团队成员分工协作提供统一的术语和接口认知。

### 1.2 项目背景

在智能制造和工业互联网场景中，设备健康状态监测和寿命预测是预测性维护（Predictive Maintenance, PdM）的核心任务。与事后维修和定期维护相比，预测性维护可以更合理地安排停机和备件计划，降低维护成本并减少突发故障带来的生产损失。

在旋转机械系统中，滚动轴承是最常见且最关键的部件之一。轴承故障通常表现为振动信号异常、温度上升、频谱结构变化、退化阶段演化等。因此，基于振动与温度等多源信息构建轴承健康预测系统，具有明确的工程价值与教学价值。

### 1.3 项目范围

本项目实现的是一个面向课程项目与实验复现的工业轴承故障预测系统，而不是单一模型脚本。系统的范围包括：

1. 真实数据集接入与统一抽象。
2. 振动信号预处理、滑动窗口和特征提取。
3. 退化阶段划分、RUL 预测、滚动预测和不确定性估计。
4. 训练过程监控、回调机制、实验记录与结果导出。
5. 指标评估与可视化输出。

系统当前不包含以下范围：

1. 在线工业边缘部署。
2. 分布式训练平台。
3. 实时流式采集系统。
4. Web 管理后台。

| RUL | Remaining Useful Life，剩余寿命 |

## 2. 总体描述

### 2.1 系统目标

系统总体目标是构建一套面向工业轴承退化分析的统一实验框架，实现以下能力：

1. 支持真实轴承寿命退化数据的导入与统一表示。
2. 支持振动信号和温度信息的预处理、切窗与特征化。
3. 支持多种建模方法进行寿命预测和状态识别。
4. 支持实验配置自动记录、训练过程监控和结果可追溯。
5. 支持课程项目所需的图表、报告和文档输出。

### 2.2 用户角色

#### 2.2.1 开发者

关注系统模块化设计、接口稳定性和算法扩展能力，希望在现有框架上快速新增模型、预处理器和评价指标。

#### 2.2.2 算法研究者

关注真实数据加载、实验可重复性、训练日志、指标和结果可视化，希望对不同模型和特征方案进行横向对比。

#### 2.2.3 课程答辩成员

关注系统是否完整、结构是否清晰、是否具备真实数据支持、是否有可展示的流程和图表输出。

### 2.3 系统运行环境

| 项目 | 要求 |
| --- | --- |
| 编程语言 | Python 3.11+ |
| 环境管理 | uv |
| 操作系统 | macOS / Linux / Windows |
| 默认硬件 | CPU 可运行 |
| 可选硬件 | 支持 CUDA 时优先使用 GPU |
| 主要依赖 | NumPy、Pandas、PyTorch、scikit-learn、Matplotlib、Seaborn 等 |

### 2.4 产品边界与假设

1. 用户具备基本 Python 环境使用能力。
2. 真实数据集由用户自行下载或由项目脚本下载到本地目录。
3. 系统以离线实验分析为主要使用方式。
4. 模型性能受数据规模、机器配置和超参数选择影响。

### 2.5 设计约束

1. 必须采用模块化结构，便于课程分工和扩展。
2. 必须保留统一命名空间 `USTC.SSE.BearingPrediction`。
3. 必须支持无真实数据时的合成数据演示。
4. 必须支持完整实验记录与可视化输出。
5. 必须提供可运行测试用例。

## 3. 业务场景与典型用例

### 3.1 场景一：基于真实数据集进行 RUL 预测

用户从 XJTU-SY 或 PHM2012 数据集中加载某个轴承实体，对振动通道进行切窗和标注，训练回归模型预测剩余寿命，并输出误差指标与预测曲线。

### 3.2 场景二：进行退化阶段划分与分类

用户基于健康指标和阶段划分策略，对轴承运行过程进行健康、早期退化、严重退化等阶段划分，并训练分类模型输出阶段标签和混淆矩阵。

### 3.3 场景三：对健康指标进行滚动预测

用户先构造健康指标序列，再训练时序模型进行单步或多步预测，用于观察退化趋势未来走向。

### 3.4 场景四：进行实验复现与报告撰写

用户需要保存模型结构、训练参数、回调配置、采样策略、指标和图表，以便后续课程答辩、论文撰写和组内复现。

## 4. 功能需求

本节给出系统必须满足的功能性需求。为便于开发和测试，每项需求均给出唯一编号。

### 4.1 数据管理需求

| 编号 | 需求说明 | 详细解释 |
| --- | --- | --- |
| FR-001 | 系统应支持 XJTU-SY 数据集目录加载 | 用户提供已解压目录后，系统应能枚举轴承实体并加载对应振动数据。 |
| FR-002 | 系统应支持 PHM2012 数据集目录加载 | 系统应支持读取 FEMTO 训练集、验证集和测试集目录。 |
| FR-003 | 系统应支持合成数据生成 | 当真实数据不可用时，系统应可生成可控退化数据用于演示和测试。 |
| FR-004 | 系统应将不同数据集统一抽象为 `BearingEntity` | 上层模块不应感知底层目录差异。 |
| FR-005 | 系统应支持枚举数据集中可用实体 | 用户可查询全部可用轴承编号。 |
| FR-006 | 系统应保留多通道信息 | 振动通道和温度通道等信息应能在实体层被访问。 |
| FR-007 | 系统应支持多种文件格式的读取与导出 | 至少支持 CSV、PKL 等常见格式。 |

### 4.2 数据预处理需求

| 编号 | 需求说明 | 详细解释 |
| --- | --- | --- |
| FR-008 | 系统应支持滑动窗口切分 | 将长振动信号切成统一长度窗口，用于模型输入。 |
| FR-009 | 系统应支持窗口步长配置 | 用户可控制重叠程度。 |
| FR-010 | 系统应支持 Z-Score 归一化 | 用于减少量纲差异影响。 |
| FR-011 | 系统应支持 Min-Max 归一化 | 便于某些模型或实验设置。 |
| FR-012 | 系统应支持鲁棒裁剪 | 用于限制异常值对模型训练的干扰。 |
| FR-013 | 系统应支持保留预处理配置 | 实验记录中必须记录窗口大小、步长和归一化方式。 |

### 4.3 特征工程需求

| 编号 | 需求说明 | 详细解释 |
| --- | --- | --- |
| FR-014 | 系统应提取时域特征 | 包括均值、方差、RMS、峰值、峭度、偏度等。 |
| FR-015 | 系统应提取频域特征 | 包括主频、频域能量、谱熵等。 |
| FR-016 | 系统应支持健康指标构造 | 供阶段划分和滚动预测使用。 |
| FR-017 | 系统应支持原始信号输入与特征输入两种表示 | 便于端到端模型与传统特征模型共存。 |

### 4.4 阶段划分需求

| 编号 | 需求说明 | 详细解释 |
| --- | --- | --- |
| FR-018 | 系统应支持 3σ 阶段划分 | 根据健康指标偏离基线的程度进行阶段判定。 |
| FR-019 | 系统应支持 FPT 阶段划分 | 根据首次出现明显可预测退化的时点划分阶段。 |
| FR-020 | 系统应输出阶段标签、边界点和健康指标 | 用于模型训练和可视化展示。 |

### 4.5 模型训练需求

| 编号 | 需求说明 | 详细解释 |
| --- | --- | --- |
| FR-021 | 系统应支持 CNN 模型 | 用于端到端振动信号建模。 |
| FR-022 | 系统应支持 RNN 模型 | 用于时序依赖建模。 |
| FR-023 | 系统应支持 Transformer 模型 | 用于序列建模和注意力可视化。 |
| FR-024 | 系统应支持 MLP 模型 | 用于特征向量建模。 |
| FR-025 | 系统应提供统一训练入口 `BaseTrainer` | 屏蔽不同模型的训练流程差异。 |
| FR-026 | 系统应提供统一测试入口 `BaseTester` | 统一输出预测结果对象。 |
| FR-027 | 系统应支持训练集与验证集分离 | 用于 EarlyStopping 与模型选择。 |
| FR-028 | 系统应支持学习率、权重衰减、批大小和最大轮次配置 | 这些参数必须可控并可记录。 |

### 4.6 回调与训练监控需求

| 编号 | 需求说明 | 详细解释 |
| --- | --- | --- |
| FR-029 | 系统应支持每个 Epoch 的自定义回调 | 回调应作为训练器扩展点。 |
| FR-030 | 系统应内置 EarlyStopping 回调 | 当验证指标无改进时提前停止训练。 |
| FR-031 | 系统应内置 TensorBoard 回调 | 输出损失、指标、梯度、参数直方图等。 |
| FR-032 | 系统应内置梯度异常报警回调 | 检测梯度消失和梯度爆炸。 |
| FR-033 | 系统应记录训练历史 | 至少记录每轮损失和关键指标。 |

### 4.7 预测方式需求

| 编号 | 需求说明 | 详细解释 |
| --- | --- | --- |
| FR-034 | 系统应支持端到端直接预测 | 输入窗口，直接输出寿命或类别。 |
| FR-035 | 系统应支持单步滚动预测 | 根据历史健康指标预测下一个时间点。 |
| FR-036 | 系统应支持多步滚动预测 | 递推生成未来多个时间点的预测值。 |
| FR-037 | 系统应支持不确定性建模 | 例如通过 MC Dropout 估计均值和不确定性。 |

### 4.8 评估与结果需求

| 编号 | 需求说明 | 详细解释 |
| --- | --- | --- |
| FR-038 | 系统应支持 MAE、MSE、RMSE、MAPE | 作为基础回归指标。 |
| FR-039 | 系统应支持 PercentError | 用于相对误差分析。 |
| FR-040 | 系统应支持 PHM2012 Score、PHM2008 Score、NASA Score | 作为挑战赛风格指标。 |
| FR-041 | 系统应支持分类准确率 | 用于阶段分类实验。 |
| FR-042 | 系统应支持结果可视化 | 至少支持预测曲线、混淆矩阵、退化阶段图和注意力热图。 |

### 4.9 实验管理需求

| 编号 | 需求说明 | 详细解释 |
| --- | --- | --- |
| FR-043 | 系统应自动生成实验目录 | 每次训练应有独立输出目录。 |
| FR-044 | 系统应自动记录实验配置 | 包括模型结构、优化器、正则化系数、采样策略、回调配置等。 |
| FR-045 | 系统应自动记录训练历史、预测结果与最终指标 | 便于复现实验。 |
| FR-046 | 系统应支持模型参数导出 | 至少支持 `pt` 检查点。 |
| FR-047 | 系统应支持 JSON/YAML/CSV/PKL 等实验产物导出 | 满足课程报告和后续分析需求。 |

## 5. 外部接口需求

### 5.1 用户接口

本项目当前主要提供命令行与 Python API 两种接口：

1. 命令行运行入口：`uv run python main.py`
2. TensorBoard 入口：`uv run tensorboard --logdir outputs/experiments`
3. Python 编程接口：`from USTC.SSE.BearingPrediction.api import *`

### 5.2 文件接口

系统输入文件包括：

1. XJTU-SY 原始 CSV 文件。
2. PHM2012 的 `acc_*.csv` 与 `temp_*.csv` 文件。
3. 用户自定义实验配置文件。

系统输出文件包括：

1. `config.json` / `config.yaml`
2. `history.csv` / `history.pkl`
3. `metrics.json` / `metrics.yaml`
4. `predictions.csv` / `predictions.pkl`
5. `alerts.json`
6. `*.pt`
7. `*.png`

### 5.3 软件接口

系统依赖的外部软件接口包括：

1. PyTorch：模型训练与推理。
2. Pandas / NumPy：数据处理。
3. Matplotlib / Seaborn：结果绘图。
4. TensorBoard：训练监控。

## 6. 数据需求

### 6.1 核心数据对象

#### 6.1.1 BearingEntity

用于统一表示一个轴承实体的全寿命快照数据，至少包含以下字段：

1. `entity_id`
2. `dataset_name`
3. `samples`
4. `sample_rate`
5. `metadata`

#### 6.1.2 BearingWindowDataset

用于统一表示切窗后的模型输入数据，至少包含以下字段：

1. `inputs`
2. `targets`
3. `metadata_frame`
4. `task_type`
5. `target_name`
6. `input_name`

### 6.2 数据质量要求

1. 快照文件应按时间顺序正确排序。
2. 振动通道应保持数值型。
3. 带表头或带附加列的 CSV 应正确兼容。
4. PHM2012 中温度文件应保留并与加速度快照对齐。

## 7. 非功能需求

### 7.1 可扩展性

系统应采用组件化设计，使新增模型、指标、回调、预处理器和阶段划分算法时无需修改主流程结构。

### 7.2 可复现性

系统应自动记录训练参数、实验配置和结果文件，确保同一实验可重复执行和对比。

### 7.3 可测试性

系统应具有自动化测试，至少覆盖：

1. 数据加载
2. 标注流程
3. 训练与回调
4. 预测方式
5. 可视化导出

### 7.4 可维护性

系统模块职责应单一明确，文档和命名应清晰，便于课程分工和后续维护。

### 7.5 易用性

系统 API 应尽可能统一，用户应能按“加载数据 -> 构造数据集 -> 训练 -> 测试 -> 评估 -> 可视化”的固定思路使用框架。

### 7.6 性能要求

1. 在 CPU 环境下应支持小规模实验可运行。
2. 在具备 GPU 条件时应自动利用 CUDA。
3. 训练日志和结果导出不应显著阻塞主训练流程。

### 7.7 可靠性要求

1. 数据缺失、文件格式不一致、空目录等情况应能给出明确错误。
2. 训练过程中的梯度异常应被记录并报警。
3. 中断后的实验目录应保留已生成结果，避免全部丢失。

---

## 8. 验收标准

系统验收时至少应满足以下条件：

1. 能成功加载 XJTU-SY 与 PHM2012 两个真实数据集。
2. 能在至少一种真实数据子集上完成训练、测试和指标输出。
3. 能输出实验配置、训练历史、模型文件和图像结果。
4. 能完成 `pytest` 自动化测试。
5. 能提供课程所需四类文档：开题报告、SRS、SDD、UML。

---

## 9. 需求总结

本系统的核心需求并不是“单个模型达到最优精度”，而是建立一套结构清晰、链路完整、对真实数据友好、对课程交付友好的工业轴承故障预测平台。需求的重点在于：

1. 真实数据支持。
2. 多模型与多任务统一框架。
3. 训练过程监控与实验管理。
4. 课程答辩所需文档、图表与可解释输出。
